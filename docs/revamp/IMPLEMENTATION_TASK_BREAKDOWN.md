# 任务细分实施文档（PostgreSQL 迁移版）

## 1. 背景与目标

本项目二次开发目标是将 `agentic-file-search` 从当前的 DuckDB + Docling + Google API 强耦合实现，演进为以下形态：

1. 仅保留 API 服务和前端交互，不再保留 CLI 主路径。
2. 数据层全量迁移到 PostgreSQL + pgvector，不走 SQLite，不保留 DuckDB 主执行路径。
3. 解析能力替换并增强：
   - PDF 使用 `pymupdf4llm`。
   - 其他格式按分格式解析器策略统一输出 Markdown。
4. 建立页级解析缓存、文档管理（CRUD）、图片语义增强缓存。
5. 模型接入统一为 OpenAI 兼容配置（文本、视觉、可选 embedding）。
6. 前端重写为 Vue3 + Vite，并由 FastAPI 单端口托管（开发和生产保持一致）。

## 2. 当前基线事实（必须作为实施前提）

1. 当前代码主路径使用 DuckDB（`src/fs_explorer/storage/duckdb.py` + `storage/__init__.py` 导出）。
2. `docker/docker-compose.yml` 中已有 PostgreSQL（pgvector 镜像）定义，但尚未被主代码调用。
3. 当前仓库依赖与脚本仍包含 CLI 路径（`pyproject.toml` 中 `explore` 脚本）。
4. 目前安装流程存在中断痕迹，`uv.lock` 可能被非预期更新，需在 M0 清理。

## 3. 实施边界与约束

1. 改造优先级：先稳定基础设施与文档，再迁移存储，再迁移接口，再迁移前端。
2. 全流程需保留 Agent + WebSocket 步骤流能力。
3. 系统安装策略：不使用虚拟环境，优先使用中国大陆镜像源（清华源）安装。
4. 文档中定义的接口契约、事件契约、表结构契约视为实施唯一标准。
5. 若某阶段未通过 DoD，不进入下一里程碑。

## M0. 环境与基线治理

### 目标

建立可重复执行的开发基线，解决中断安装与依赖源问题，冻结“改造前事实”。

### 前置依赖

1. 仓库已拉取并在目标分支。
2. 具备系统 Python 可写权限（或可执行用户级安装）。

### 任务清单（可执行）

1. 终止残留安装进程，清理中断安装状态。
2. 恢复 `uv.lock` 到仓库基线版本（若当前有脏变更）。
3. 配置系统安装镜像：
   - `uv pip --system` + 清华源；
   - fallback 到 `pip` 清华源。
4. 运行基线检查并记录结果：
   - `pytest`
   - `ruff check`
5. 输出《基线检查记录》：
   - 成功项
   - 失败用例
   - 环境信息（Python、uv、OS、镜像源）

### 完成定义（DoD）

1. 基线命令可执行，且日志可追溯。
2. `uv.lock` 状态明确（已恢复或变更原因明确记录）。
3. 后续成员可按文档在同类环境复现安装步骤。

### 风险与兜底

1. 风险：系统环境权限不足导致安装失败。
2. 兜底：切换 `--user` 安装并记录 PATH 调整步骤。

### 产出物

1. 基线检查记录（Markdown）。
2. 安装与镜像源说明（追加到 README 迁移章节草稿）。

## M1. 存储层全量迁移（DuckDB -> PostgreSQL + pgvector）

### 目标

将存储、检索、向量能力全部迁移到 PostgreSQL，并保留上层调用语义。

### 前置依赖

1. M0 完成。
2. PostgreSQL（pgvector）可用。

### 任务清单（可执行）

1. 新建 PostgreSQL 存储实现（替换 DuckDB 主路径）。
2. 设计并落地核心表：
   - `documents`
   - `parsed_units`
   - `image_semantics`
   - `chunks`（检索支撑）
3. 设计并执行 DDL 与索引脚本（含 pgvector 索引）。
4. 替换 `search/query` 对 DuckDB 的专用逻辑。
5. 替换 `indexing/pipeline` 的写入与增量维护逻辑。
6. 保留接口层抽象，避免业务层硬编码数据库驱动细节。

### 完成定义（DoD）

1. DuckDB 不再作为主执行路径。
2. 关键词、元数据过滤、向量检索在 PostgreSQL 上可跑通。
3. 关键查询有索引策略并能通过基本性能验证。

### 风险与兜底

1. 风险：向量检索性能波动。
2. 兜底：先以候选集重排保证正确性，再逐步优化索引/参数。

### 产出物

1. PostgreSQL 存储实现。
2. DDL/迁移脚本。
3. 存储层测试用例与性能基线记录。

## M2. 解析与缓存链路升级

### 目标

建立“按页解析 + 增量缓存 + 图片语义延迟增强”的稳定数据通路。

### 前置依赖

1. M1 完成。
2. 文档上传托管路径可用。

### 任务清单（可执行）

1. 解析器策略落地：
   - PDF：`pymupdf4llm(page_chunks=True)`。
   - DOCX/PPTX/XLSX/HTML/MD：分格式解析器输出统一 Markdown。
   - DOC：LibreOffice headless 转 DOCX 后复用。
2. 页级缓存策略：
   - `parsed_units` 按 `document_id + page_no + parser_version` 幂等更新。
   - 基于内容 hash 判断增量重解析。
3. 图片处理策略：
   - 解析阶段提取图片并计算 `image_hash`。
   - `image_semantics` 写入基础信息，语义字段可为空。
4. Lazy 语义增强：
   - 问答命中相关页时触发视觉模型增强；
   - 完成后按 `image_hash` 回填语义，重复命中不重复调用。

### 完成定义（DoD）

1. 同文档重复解析可命中缓存，且仅重解析变更页。
2. 图片语义增强具备幂等行为（同 hash 只增强一次）。
3. 解析失败路径可返回明确错误与定位信息。

### 风险与兜底

1. 风险：DOC 转换依赖缺失。
2. 兜底：返回明确可操作错误（提示安装 LibreOffice）。

### 产出物

1. 解析器适配层。
2. 缓存逻辑与图片增强逻辑。
3. 对应单测与集成测试。

## M3. API 与 WebSocket 协议扩展

### 目标

提供完整文档管理与问答接口，并扩展 WS 事件协议支持缓存与图片增强状态。

### 前置依赖

1. M2 完成。

### 任务清单（可执行）

1. 文档 CRUD API：
   - 上传托管、列表、详情、更新、删除。
2. 解析 API：
   - 触发解析（全量/增量）；
   - 分页读取解析结果。
3. 问答 API：
   - 保留 Agent 逻辑与引用输出。
4. WS 事件扩展：
   - `cache_hit`
   - `image_enhance_started`
   - `image_enhance_done`
5. 错误码与可观测字段标准化（trace_id、error_code、message）。

### 完成定义（DoD）

1. API 契约字段稳定，能直接生成测试用例。
2. WS 事件可覆盖完整会话生命周期。
3. 错误码体系可定位主要故障场景。

### 风险与兜底

1. 风险：协议变更导致前端不兼容。
2. 兜底：事件 payload 增量兼容，保留既有字段。

### 产出物

1. API 路由实现与契约文档。
2. WS 事件协议实现与示例。

## 5. 变更影响矩阵（模块级）

| 模块 | 影响级别 | 主要变更 | 风险等级 | 备注 |
|---|---|---|---|---|
| `storage` | 高 | DuckDB 实现替换为 PostgreSQL + pgvector | 高 | 核心基础设施 |
| `search` | 高 | 查询 SQL、向量检索、排序融合迁移 | 高 | 影响问答质量 |
| `indexing` | 高 | 写入与增量更新策略迁移 | 高 | 影响缓存和新鲜度 |
| `server` | 高 | 新 API、WS 事件扩展、静态托管 | 中 | 影响前后端联动 |
| `agent` | 中 | 工具链与模型配置改造 | 中 | 影响可用性与成本 |
| `tests` | 高 | 大量重写与新增契约测试 | 中 | 质量门禁 |

## 6. 统一验收标准

1. 功能验收：
   - 文档 CRUD、解析、问答、引用、WS 步骤流均可用。
2. 架构验收：
   - PostgreSQL 成为唯一主存储与检索路径（DuckDB 退出主链路）。
3. 性能验收：
   - 页级缓存生效，重复解析有明显命中。
4. 可运维验收：
   - 单端口部署清晰、日志可追踪、错误可定位。
5. 文档验收：
   - README、配置说明、迁移手册、回滚手册齐备。

## 7. 执行顺序与门禁策略

1. M0 -> M1 -> M2 -> M3 -> M4 -> M5 严格顺序执行。
2. 每个里程碑通过 DoD 审核后方可进入下一阶段。
3. 发现关键阻塞时，先回填风险登记和兜底措施，再继续推进。
